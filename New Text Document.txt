Title: No Auto-Failover? No Problem — A Service Discovery Pattern for Confluent Cloud DR

Confluent Cloud does not support automatic DR failover or failback. For enterprises running mission-critical event streaming, this means a cluster switchover can trigger a wave of manual application reconfigurations — each team hunting down a new broker URL and redeploying. At scale, that is not a DR strategy. It is a crisis.

In this lightning talk, we share how our platform team at ABC solved this with a service discovery pattern built on Azure Key Vault. Our CI/CD pipelines handle the full DR lifecycle — failover and failback — and on every switchover, the active cluster bootstrap URL is automatically written to Key Vault. Applications resolve the correct broker endpoint at startup with no configuration changes, no manual intervention, and no dependency on Confluent providing a unified broker URL.

We will cover the pattern architecture, the CI/CD pipeline design, how we handle the one remaining application touchpoint (a restart to refresh cluster metadata), and what we learned running this in production across a multi-program enterprise deployment.

Key Takeaways:
• How to implement automated DR failover and failback for Confluent Cloud using CI/CD pipelines
• Using Azure Key Vault as a dynamic broker URL registry to eliminate operational burden on application teams
• Tradeoffs and real-world lessons from production — what worked, what we would do differently
• A replicable pattern any team running multi-cluster Confluent Cloud deployments can adopt
